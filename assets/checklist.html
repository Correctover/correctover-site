Use this checklist to assess your AI runtime security posture. Check off completed items, add notes, then download as PDF to share with your team.

1
Runtime Attack Surface
4 items
01.RCE detection in model inference pipeline
Are model inference endpoints protected against remote code execution via malicious inputs or deserialization?
02.SSRF protection for internal service calls
Can model-serving components be abused to scan internal networks or access cloud metadata endpoints?
03.Cloud credential exfiltration monitoring
Is there real-time detection for credentials being leaked through model outputs, logs, or error messages?
04.Path traversal in file I/O operations
Are file read/write operations in the AI pipeline guarded against directory traversal attacks?
2
AI Framework Security
4 items
05.MCP tool call validation and sandboxing
Are MCP (Model Context Protocol) tool invocations validated and sandboxed to prevent unauthorized actions?
06.Agent orchestration pipeline integrity
Is the agent orchestration pipeline protected against prompt injection and unauthorized workflow manipulation?
07.Deserialization attack prevention
Are pickle, joblib, or other serialization formats in the ML pipeline protected against arbitrary code execution?
08.Training data exfiltration detection
Can the system detect unauthorized extraction of training data through model responses or side channels?
3
Compliance Readiness
4 items
09.CCS standard alignment (24 detection rules)
Does your detection pipeline map to Correctover CCS (Cloud Security Standard) with all 24 CRITICAL/HIGH rules?
10.SOC2 evidence collection automation
Is evidence collection for SOC2 Type II audits automated across AI runtime controls?
11.Audit trail completeness verification
Are all AI runtime access events, model inferences, and configuration changes captured in immutable audit trails?
12.Incident response time measurement
What is your measured mean time to detect (MTTD) and mean time to respond (MTTR) for AI security incidents?
4
Detection Coverage
4 items
13.Real-time vs batch detection capability
Does your detection run real-time (sub-second) on inference traffic, or only as batch analysis of logs?
14.False positive rate benchmarking
What is the measured false positive rate (FPR) of your detection rules under production traffic?
15.Detection rule update frequency
How frequently are detection rules updated for new AI-specific attack vectors (supply chain, plugin exploits)?
16.Custom rule support and extensibility
Can your security team author custom detection rules specific to your AI workload without vendor lock-in?
5
Deployment & Operations
4 items
17.Deployment complexity (time to first detection)
How long does it take from deployment start to first AI-specific security detection? Hours? Days? Weeks?
18.Performance overhead (latency impact P50/P99)
What is the measured latency overhead at P50 and P99 for inline detection on inference traffic?
19.Integration with existing SIEM/SOC
Does the solution integrate with your existing SIEM (Splunk, Elastic, Sentinel) and SOC workflows via standard formats?
20.Multi-cloud and hybrid environment support
Does the detection coverage extend across AWS, Azure, GCP, and on-premise AI workloads consistently?
6
Vendor Assessment
4 items
21.Security team responsiveness (SLA)
What is the SLA for security incident response from your vendor's security team? Is 24/7 coverage available?
22.Public vulnerability disclosure track record
Does the vendor have a published vulnerability disclosure policy and a track record of responsible disclosure?
23.Independent security audit results
Has the vendor undergone independent third-party security audits? Are results available under NDA?
24.Third-party integrations and ecosystem
Does the solution integrate with your existing toolchain (LangChain, LlamaIndex, MCP servers, Docker, Kubernetes)?
