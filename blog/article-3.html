CCS Compliance for AI Systems: A 24-Rule Framework for Production Security

By Correctover Security TeamJuly 20, 202610 min read

Security compliance frameworks are playing catch-up with AI. NIST CSF 2.0 added a brief mention of AI risks in its February 2024 update. SOC 2’s trust services criteria were never designed with LLM agents in mind. ISO 27001 controls cover general information security but miss the specific attack surfaces introduced by AI inference pipelines, agentic tool execution, and model supply chains.

This gap is not theoretical. Every RCE vulnerability we’ve disclosed across frameworks like CrewAI, AutoGen, and LangGraph would pass a SOC 2 Type II audit. Why? Because existing frameworks audit processes and policies, not runtime behavior of AI agent execution paths.

The Cloud Cybersecurity Standard (CCS) for AI Systems was designed to fill this gap. This article explains the 24-rule framework at a high level, how it differs from traditional compliance frameworks, and how Correctover maps detection events to CCS rules automatically.

What Is CCS?

CCS (Cloud Cybersecurity Standard) is a risk-based security framework purpose-built for AI workloads in production. Unlike general-purpose frameworks, CCS focuses on the runtime attack surface: the moment when model output becomes executable action.

The framework organizes 24 detection rules into 6 categories, each mapping to a distinct phase of the AI agent lifecycle:

RCE — Remote Code Execution Prevention (6 rules)

Detects shell injection, code evaluation, and unsafe subprocess patterns in tool call arguments, agent messages, and code generation outputs.

CCS-RCE-001 through CCS-RCE-006

SSRF — Server-Side Request Forgery Prevention (4 rules)

Validates model provider endpoints, tool webhook URLs, and document ingestion sources against internal/reserved IP ranges and known-bad domains.

CCS-SSRF-001 through CCS-SSRF-004

DESER — Unsafe Deserialization Prevention (4 rules)

Identifies pickle opcodes, yaml.load() patterns, and Java serialization gadget chains in state serialization, document loading, and cross-node communication.

CCS-DESER-001 through CCS-DESER-004

CRED — Credential and Secret Exfiltration Prevention (4 rules)

Monitors model responses, tool return values, and agent output for patterns matching API keys, tokens, database connection strings, and cloud provider secrets.

CCS-CRED-001 through CCS-CRED-004

INJECT — Prompt and Command Injection Detection (4 rules)

Detects prompt injection payloads, jailbreak patterns, and command injection attempts in user-facing agent inputs and chained model calls.

CCS-INJECT-001 through CCS-INJECT-004

DRIFT — Agent Pipeline Integrity (2 rules)

Detects unauthorized modification of agent pipelines, unexpected tool registrations, and graph topology changes that may indicate supply chain compromise.

CCS-DRIFT-001 through CCS-DRIFT-002

Why Traditional Frameworks Fall Short

To understand the gap, consider how a SOC 2 audit approaches a typical AI deployment:

Control AreaSOC 2 CriterionAI Blind Spot
Access Control CC6.1: Logical and physical access Does not cover tool-level access control. An agent with valid API credentials can execute arbitrary tool calls, and SOC 2 doesn’t inspect tool call arguments.
Change Management CC8.1: Changes to system components Does not cover agent pipeline drift. A prompt change that introduces RCE risk through tool call injection is not captured by traditional change management.
Risk Mitigation CC7.1: Identify and respond to threats Does not cover model-output-to-execution risk. The threat model assumes human-mediated command execution, not autonomous agent tool use.
Monitoring CC7.2: Monitor system operations Logging agent actions is not the same as detecting malicious tool calls. Standard SIEM integration misses the semantic layer of agent behavior.

NIST CSF 2.0 fares slightly better with its new “Govern” function (GV.OC, GV.SC), which acknowledges AI-specific risk. But NIST remains a high-level framework — it recommends that organizations “manage AI risks” but doesn’t specify what detection rules should look like at the runtime level.

CCS bridges this gap by providing prescriptive, testable detection controls for the AI runtime layer. Each CCS rule is:

  • Machine-testable — Can be verified via automated probe, not manual attestation
  • Framework-agnostic — Works across CrewAI, AutoGen, LangGraph, Haystack, and any other framework
  • Evidenced — Each rule is backed by a confirmed vulnerability disclosure (see our bounty-to-defense article)
  • Latency-budgeted — Designed to execute within 15 microseconds so security doesn’t degrade user experience

CCS Adoption Path

Implementing CCS compliance typically follows three phases:

Phase 1: Monitor (Days 1–3)

Deploy Correctover in monitor-only mode. All 24 CCS rules run against production traffic but do not block. This establishes a baseline: how many detections per day, which rules trigger, and whether any false positives appear in your specific traffic patterns.

Phase 2: Enforce (Days 4–10)

After reviewing the Phase 1 baseline, enable enforcement. Critical and High severity rules (CCS-RCE, CCS-SSRF, CCS-DESER categories) block automatically. Rules in the INJECT and DRIFT categories remain in monitor mode for further tuning.

Phase 3: Audit (Day 11+)

Export the CCS compliance report for internal audit or third-party review. The report provides machine-verifiable evidence that all 24 rules are active and have been tested against production traffic.

Beyond Compliance: Continuous Validation

One of the limitations of traditional compliance frameworks is the annual audit cycle. A SOC 2 report from six months ago tells you nothing about your security posture today, especially in the fast-moving AI ecosystem where new frameworks and attack surfaces emerge weekly.

CCS is designed for continuous validation. Because Correctover runs inline with production traffic, every request is a compliance check. The CCS report updates in real time, and any degradation (a rule that stops triggering due to framework updates) is detected immediately.
